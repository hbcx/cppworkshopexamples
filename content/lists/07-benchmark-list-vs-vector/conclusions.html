The five things worth taking away

  1. O(1) insertion is real, and it is not where the time goes. Both containers must find the position first, and that search is O(n) for both. The list wins the cheap half of the operation and loses the expensive half, because its search chases a pointer into a fresh cache line per element while the vector's walks straight through memory the prefetcher has already fetched.
  2. Shifting the tail is nearly free; the analysis says it is the expensive part. The vector's O(n) shift is a memmove of trivially copyable integers — wide vector loads and stores, no branches, perfectly predictable. Complexity counts it as n operations. The hardware charges for roughly one cache line per 16 of them.
  3. The gap widens with n, which is the opposite of what the exponents predict. 4.1× at a hundred elements, 18.8× at ten thousand. As the list outgrows the cache its per-hop cost rises, while the vector's stays flat — the traversal table shows this in isolation: 1.0 ns to 3.6 ns per element for the list, an unmoving 0.2 ns for the vector.
  4. Read the traversal row honestly. Part of that 0.2 ns is the compiler vectorizing a contiguous sum into SIMD, an optimization the list cannot receive at any n. That is not a measurement artefact to correct for — it is a permanent structural advantage of contiguous storage, and it is fair to count it.
  5. None of this makes the list useless — it makes complexity a bad reason to pick it. The list's case was never speed. It is the guarantee that nodes never move: references stay valid across every mutation, and splice transfers elements between containers in O(1) without copying, moving, or even requiring the type to be movable. A vector cannot do those things at any speed. Those are the reasons; the asymptotics are not.

Rule of thumb. Default to std::vector, including when the complexity table says otherwise. Reach for std::list when you need stable references into a container that keeps changing, or when splice is doing something no other container can — and measure if you believe it will also be faster.
