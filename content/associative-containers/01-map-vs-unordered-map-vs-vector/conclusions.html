The five things worth taking away

  1. Hashing wins lookups, and wins them wider the more data you have. The complexity column already told you that, and the measurement agrees: 1.3× ahead of the tree at a hundred keys, 20.9× ahead at a million.
  2. O(1) is a statement about comparisons, not about time. unordered_map still grew 9.3× across the range while doing the same single bucket probe throughout. That growth is not algorithmic — it is the memory hierarchy. Complexity analysis assumes every memory access costs the same; on real hardware a miss to DRAM costs roughly a hundred times a hit in L1, and there is no exponent anywhere in the analysis that accounts for it.
  3. At equal complexity, layout decides. The sorted vector has exactly the same O(log n) as std::map and performs the same number of comparisons — yet it was 5.1× faster at a million keys. A binary search walks contiguous memory that the prefetcher can see coming; a tree chases a pointer into a fresh cache line at every level. Same algorithm, different hardware behaviour.
  4. The vector was the cheapest to build at every single size — node allocation being what it is. Read that honestly, though: it is cheap to build once. Every later insertion costs O(n), which is precisely why the sorted vector belongs to lookup tables that are filled and then only read.
  5. What the numbers cannot show you is often what decides. unordered_map has no ordered iteration and no lower_bound at all. If you need a range query or sorted traversal, the fastest container is simply not on the table, and no benchmark will tell you so.

Rule of thumb. Reach for unordered_map for plain key/value lookup. For a read-mostly table, sort a vector and binary-search it. Reach for std::map when you genuinely need the ordering — and now you know what it costs.
